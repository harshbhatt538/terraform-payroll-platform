## Task 3 - Security & Access Control

### 3a. IAM & Role-Based Access Control

Three separate IAM roles are defined in `terraform/modules/iam/` - one per tenant type
(companies, bureaus, employees). Each role follows least-privilege strictly:

- **S3 access** is scoped to the tenant's own prefix only (`companies/*`, `bureaus/*`,
  `employees/*`). An explicit `Deny` statement blocks access to other tenant prefixes -
  this means even if application logic sends a wrong request, IAM rejects it at the
  AWS layer.

- **Secrets Manager access** is scoped by path - the companies role can only read secrets
  under `test/companies/*` and cannot read bureau or employee secrets under
  any condition.

- **No wildcard resource access** exists anywhere in the IAM configuration. Every policy
  targets a specific ARN or ARN pattern.

- **No hardcoded credentials** exist anywhere in the Terraform code. The RDS master
  password is generated by the `random_password` resource and stored directly in
  Secrets Manager - it never appears in code, logs, or state files in plain text.

Each EC2 instance is assigned only its own tenant's instance profile. A companies EC2
cannot assume a bureaus role or access bureaus resources at the AWS API level.

---

### 3b. Secrets Management

All sensitive values are managed through AWS Secrets Manager:

- The RDS master password is generated at `terraform apply` time using the `random_password`
  resource (32 characters, special characters included) and stored in Secrets Manager
  at path `test/database/master-password-{environment}`.

- Each tenant's application secrets are stored under their scoped path:
  - `test/companies/*`
  - `test/bureaus/*`
  - `test/employees/*`

**Runtime secret injection** - secrets are never passed as environment variables or
written to config files. At application startup, the service calls Secrets Manager
directly using the EC2 instance's IAM role credentials (via IMDSv2). See
`docs/secret_injection_example.py` for the reference implementation.

Key points:
- `DB_SECRET_NAME` is injected as an environment variable - only the secret *name*,
  never the secret *value*
- Secrets are fetched in memory at runtime and never logged
- IMDSv2 is enforced on all EC2 instances (`http_tokens = "required"`) - this prevents
  SSRF attacks from stealing instance credentials via the metadata endpoint

### 3c. Encryption

**Encryption at rest:**

| Resource | Method | Config Location |
|----------|--------|----------------|
| RDS | `storage_encrypted = true` (AES-256) | `modules/database/main.tf` |
| S3 | Server-side encryption AES-256 | `modules/storage/main.tf` |
| EC2 root volumes | `encrypted = true` on root block device | `modules/compute/main.tf` |

**Encryption in transit:**

- **RDS** - SSL is enforced via a custom parameter group with `rds.force_ssl = 1`.
  Any connection attempt without SSL is rejected at the database level, not just
  discouraged.

- **S3** - A bucket policy `DenyNonHTTPS` statement rejects any request where
  `aws:SecureTransport = false`. This means plain HTTP requests to the bucket are
  denied regardless of IAM permissions.

- **EC2 to RDS** - All application code connects to RDS over SSL. The RDS endpoint
  only accepts connections from the three tenant security groups on port 5432,
  and `rds.force_ssl` ensures those connections must be encrypted.

- **EC2 to Secrets Manager / AWS APIs** - All AWS SDK calls use HTTPS by default.
  The NACL outbound rules allow port 443 only, enforcing this at the network layer.

---

### 3d. Network Security

**Security Groups:**

Each tenant EC2 instance has its own security group that:
- Accepts inbound traffic only on port 8080 from within the VPC CIDR
- Has no inbound rule allowing direct internet access
- Allows outbound only on ports 443 (HTTPS) and 5432 (PostgreSQL)
- Has no rule permitting traffic to or from another tenant's security group

The RDS security group:
- Accepts inbound only on port 5432
- Only from the three tenant EC2 security groups - not from the VPC CIDR broadly
- Has no public inbound rule of any kind

**NACLs:**

Private subnet NACLs (defined in `modules/vpc/main.tf`) restrict traffic to:

| Direction | Port | Reason |
|-----------|------|--------|
| Inbound | 8080 | Application traffic from VPC only |
| Inbound | 5432 | PostgreSQL from VPC only |
| Inbound | 1024–65535 | Ephemeral ports for return traffic |
| Outbound | 443 | HTTPS to AWS APIs |
| Outbound | 5432 | PostgreSQL to RDS |
| Outbound | 1024–65535 | Ephemeral return traffic |

> Ephemeral ports (1024–65535) are required because when a server initiates an outbound
> TCP connection, the OS assigns a random high port for the return traffic. Without this
> rule, response packets are dropped by the NACL even though the connection was legitimate.

**Database and internal service accessibility:**

- RDS has `publicly_accessible = false` - it has no public endpoint
- All EC2 instances have `associate_public_ip_address = false` - no public IPs assigned
- All instances live in private subnets - not routable from the internet directly
- Only the NAT Gateway in the public subnet allows outbound internet access from
  private subnets - no inbound path exists from internet to private subnet

**Cross-tenant traffic prevention:**

Even though all three tenant EC2 instances share the same VPC, cross-tenant traffic
is prevented at two layers:

1. **Security Group layer** - no security group has an ingress rule referencing another
   tenant's security group. A companies EC2 cannot open a connection to a bureaus EC2
   on any port.

2. **IAM layer** - even if a network path existed, the IAM role on each EC2 instance
   cannot access another tenant's S3 prefix, Secrets Manager path, or AWS resources.
   Two independent enforcement boundaries exist for every access boundary.
